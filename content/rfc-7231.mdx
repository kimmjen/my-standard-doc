# HTTP/1.1 Semantics and Content

[RFC 7231](https://datatracker.ietf.org/doc/html/rfc7231)은 HTTP/1.1 프로토콜의 **의미(Semantics)와 내용(Content)**을 정의하는 가장 기초적인 문서입니다.
개발자가 API를 설계할 때 가장 많이 고민하는 **HTTP 메서드**와 **상태 코드**의 기준이 여기에 담겨 있습니다.

---

## 1. HTTP 메서드 (Request Methods)

리소스에 대해 수행하길 원하는 행동을 나타냅니다.

### 주요 메서드
*   **GET**: 리소스 조회. 서버의 상태를 변경하지 않는 **안전한(Safe)** 메서드여야 합니다.
*   **POST**: 리소스 생성 또는 데이터 처리. (예: 글 쓰기, 결제 요청)
*   **PUT**: 리소스의 **전체** 수정 (덮어쓰기). 만약 리소스가 없으면 생성할 수도 있습니다.
*   **DELETE**: 리소스 삭제.
*   **PATCH**: 리소스의 **부분** 수정. (참고: PATCH는 RFC 5789에서 정의됨)

> [!TIP]
> **Idempotent (멱등성)이란?**
> 동일한 요청을 한 번 보내든, 여러 번 연속으로 보내든 서버의 상태 결과가 똑같아야 한다는 성질입니다.
> *   **멱등함**: GET, PUT, DELETE
> *   **멱등하지 않음**: POST (계속 보내면 글이 계속 생김)

## 2. 상태 코드 (Status Codes)

서버가 요청을 처리한 결과를 3자리 숫자로 알려줍니다.

### 2xx: 성공 (Success)
*   `200 OK`: 요청 성공. (가장 일반적)
*   `201 Created`: 리소스 생성 성공. (POST 요청 시 주로 사용)
*   `204 No Content`: 성공했지만 응답 본문(Body)에 줄 데이터가 없음. (삭제 성공 등)

### 3xx: 리다이렉션 (Redirection)
*   `301 Moved Permanently`: 주소가 영구적으로 바뀜. (SEO 점수 이동)
*   `302 Found`: 주소가 잠시 바뀜.
*   `304 Not Modified`: 캐시된 데이터가 변경되지 않았음. (클라이언트 캐시 재사용)

### 4xx: 클라이언트 오류 (Client Error)
*   `400 Bad Request`: 요청 파라미터가 잘못됨.
*   `401 Unauthorized`: 인증(로그인)이 필요함.
*   `403 Forbidden`: 권한이 없음. (로그인은 했으나 접근 불가)
*   `404 Not Found`: 리소스를 찾을 수 없음.
*   `405 Method Not Allowed`: 해당 리소스에서 이 메서드를 지원하지 않음.

### 5xx: 서버 오류 (Server Error)
*   `500 Internal Server Error`: 서버 내부에서 알 수 없는 에러 발생.
*   `502 Bad Gateway`: 게이트웨이(프록시)가 잘못된 응답을 받음.
*   `503 Service Unavailable`: 서버 과부하 또는 점검 중.

---

## 3. 컨텐츠 협상 (Content Negotiation)

클라이언트와 서버가 서로 이해할 수 있는 데이터 형식을 맞추는 과정입니다.

*   `Accept`: 클라이언트가 어떤 형식을 원하는지 (예: `application/json`)
*   `Content-Type`: 보내는 데이터가 어떤 형식인지 (예: `text/html`)
