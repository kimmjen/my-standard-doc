# HTTP Semantics

[RFC 9110](https://datatracker.ietf.org/doc/html/rfc9110)은 HTTP 프로토콜의 **최신 통합 명세**입니다.
기존 RFC 7231을 대체하며, HTTP/1.1, HTTP/2, HTTP/3 모두에 적용되는 핵심 의미론을 정의합니다.

---

## 1. RFC 7231과의 관계

2022년 6월, IETF는 HTTP 관련 문서를 현대화하여 재구성했습니다.

| 기존 (RFC 723x 시리즈) | 신규 (RFC 9xxx 시리즈) |
|----------------------|----------------------|
| RFC 7230 (메시지 구문) | RFC 9112 (HTTP/1.1) |
| RFC 7231 (의미론) | **RFC 9110 (핵심 의미론)** |
| RFC 7232 (조건부 요청) | RFC 9110에 통합 |
| RFC 7235 (인증) | RFC 9110에 통합 |

## 2. HTTP 메서드 정의

### Safe Methods (안전한 메서드)
서버의 상태를 변경하지 않는 메서드입니다.

| 메서드 | 용도 |
|--------|------|
| `GET` | 리소스 조회 |
| `HEAD` | 헤더만 조회 (응답 본문 없음) |
| `OPTIONS` | 지원하는 메서드 확인 |
| `TRACE` | 루프백 테스트 |

### Idempotent Methods (멱등 메서드)
여러 번 실행해도 결과가 같은 메서드입니다.

| 메서드 | 설명 |
|--------|------|
| `PUT` | 리소스 전체 교체 |
| `DELETE` | 리소스 삭제 |
| + 모든 Safe 메서드 | |

### 그 외
| 메서드 | 특징 |
|--------|------|
| `POST` | 비멱등, 새 리소스 생성 |
| `PATCH` | 비멱등, 부분 수정 |

---

## 3. 상태 코드 분류

```
1xx: 정보 응답 (처리 중)
2xx: 성공
3xx: 리다이렉션
4xx: 클라이언트 오류
5xx: 서버 오류
```

### 자주 사용하는 상태 코드

| 코드 | 의미 | 사용 시점 |
|------|------|----------|
| 200 | OK | 성공적인 GET/PUT/PATCH |
| 201 | Created | POST로 리소스 생성 시 |
| 204 | No Content | DELETE 성공, 본문 없음 |
| 400 | Bad Request | 잘못된 요청 형식 |
| 401 | Unauthorized | 인증 필요 |
| 403 | Forbidden | 권한 없음 |
| 404 | Not Found | 리소스 없음 |
| 409 | Conflict | 충돌 (중복 등) |
| 422 | Unprocessable | 유효성 검증 실패 |
| 500 | Internal Error | 서버 내부 오류 |

---

## 4. 콘텐츠 협상 (Content Negotiation)

클라이언트가 원하는 형식을 요청하면 서버가 맞춰서 응답합니다.

```http
GET /users/123 HTTP/1.1
Accept: application/json
Accept-Language: ko-KR
```

| 헤더 | 용도 |
|------|------|
| `Accept` | 원하는 미디어 타입 |
| `Accept-Language` | 원하는 언어 |
| `Accept-Encoding` | 원하는 압축 방식 |

---

> [!TIP]
> **실무 권장**  
> API 설계 시 RFC 9110의 메서드와 상태 코드 의미를 정확히 따르면, 다른 개발자도 문서 없이 API 동작을 예측할 수 있습니다.
