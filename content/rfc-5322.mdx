# Email Syntax (RFC 5322)

우리가 흔히 쓰는 이메일 주소(`user@example.com`)의 형식을 정의한 표준입니다.
회원가입 폼에서 "이메일 형식이 올바르지 않습니다"라는 에러를 띄울 때, 그 기준이 되는 문서입니다.

---

## 1. 정규식(Regex)의 함정

인터넷에 돌아다니는 "이메일 검증 정규식"의 99%는 **RFC 표준을 완벽하게 지키지 않습니다.**
왜냐하면 표준이 허용하는 이메일 형식이 상상 이상으로 기괴하기 때문입니다.

### 표준상 가능한 이메일들 (충격 주의)
*   `"very.(),:;<>[]\".VERY.\"very@\\ \"very\".unusual"@strange.example.com`
*   `user@[IPv6:2001:db8::1]`

개발자가 완벽한 정규식을 짜려고 노력하다 보면 코드가 100줄이 넘어가게 됩니다.

---

## 2. 실무적인 검증 방법

그래서 실무에서는 복잡한 정규식으로 완벽 검증을 하려 하기보다, **"실용적인 검증"**을 택합니다.

### 1단계: 최소한의 형식 확인 (HTML5)
HTML5의 `<input type="email">`이나 간단한 정규식으로 `@`가 있는지, 도메인 점(`.`);이 있는지 정도만 봅니다.
```javascript
// 복잡한 검증보다는 이 정도가 현실적입니다.
const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
```

### 2단계: 이메일 발송 확인 (Verification Email)
**가장 확실한 방법**입니다.
가입 시점에 사용자가 입력한 이메일로 **"인증 링크"**를 보내고, 클릭하게 만듭니다.
이메일이 실제로 존재하고 주인이 맞는지 확인하는 유일한 방법입니다.

---

## 3. Plus Addressing (Sub-addressing)
Gmail 기능으로 유명한 `user+tag@example.com` 형식입니다.
*   `+` 뒤의 문자는 무시하고 같은 메일함으로 들어옵니다.
*   개발자는 이 `+` 기호를 **특수문자 에러**로 처리하면 안 됩니다. 표준에 맞는 올바른 주소입니다.
